# Tutorio — анализ идеи и план разработки веб-MVP

## Контекст

Продукт: «финансовый календарь» для частных преподавателей и малых школ (до 3–5 преподавателей). Ядро — абонементы, баланс занятий и журнал операций (ledger), а не просто CRM/календарь. Первый пилот — собственная школа SpeakWise.

Решения, принятые с пользователем:
- **Стек**: монорепозиторий, отдельный **NestJS API + Next.js web** (как в исходном плане пользователя).
- **Языки UI**: украинский + английский с самого начала (i18n-каркас обязателен).
- **Оплаты в MVP**: только ручная фиксация; онлайн-эквайринг — следующий этап, но интерфейс `PaymentProvider` закладываем сразу.
- **Режим разработки**: один разработчик, параллельно с основной работой → план строится вертикальными срезами, горизонт ~14–16 недель.
- Мобильное приложение (Expo) — после веб-MVP, вне этого плана.

## Оценка исходного плана

Сильные стороны (сохраняем как есть): позиционирование через сложные финансовые кейсы; ledger вместо мутируемого `balance`; фиксация цены на момент покупки; `LessonSeries` ≠ `Lesson`; деньги в minor units; workspace-модель; отказ от приёма денег на свой счёт; чёткий список анти-скоупа.

### Что доработать в плане (пробелы, которые нужно учесть сразу)

1. **Два разных журнала, не один.** Денежный журнал (Payment, в валюте, minor units) и журнал урочных кредитов (LessonCreditEntry, в штуках занятий) — разные сущности с разной семантикой. Смешивать их в один `LedgerEntry` нельзя: «+8 занятий» и «+200 EUR» — операции в разных единицах. Связь между ними: покупка пакета = Payment (деньги) + начисление кредитов (уроки).
2. **Таймзоны и DST.** Повторяющееся расписание хранится как правило в таймзоне преподавателя (`weekday + local time + tz`), конкретные `Lesson` материализуются в UTC. Иначе при переходе на летнее время все повторяющиеся уроки «съедут» на час. У учеников своя таймзона — время показывается каждому в его зоне.
3. **Материализация серий.** `Lesson` генерируются из `LessonSeries` на скользящий горизонт (например, 12 недель вперёд) cron-задачей. Редактирование серии — с семантикой «только этот урок / этот и последующие» (как в Google Calendar). Перенесённый/отменённый урок помечается `isDetached` и не перегенерируется.
4. **State machine статуса урока.** Явные статусы и разрешённые переходы: `scheduled → completed | cancelled_by_teacher | cancelled_by_student | no_show`; отмена учеником делится на «вовремя» и «поздно» по конфигурируемому дедлайну (часов до урока). Каждый переход порождает (или не порождает) операцию в журнале кредитов — это ядро продукта, оно живёт в `packages/domain` и покрывается юнит-тестами в первую очередь.
5. **Идемпотентность операций журнала.** Уникальный ключ вида `(lessonId, entryType)` на списание/возврат — защита от двойного списания при повторном клике или ретрае. Смена статуса урока назад (completed → scheduled) создаёт компенсирующую запись, а не удаляет старую.
6. **Пакет принадлежит Enrollment, а не группе.** В группе у каждого ученика свой пакет, своя цена, своя дата оплаты. Все финансовые сущности привязаны к `Enrollment` (связка студент+группа или студент+индивидуально).
7. **Импорт из таблиц (CSV).** Критичен и для переноса SpeakWise, и для онбординга любого преподавателя — все они уходят с Google Sheets. В исходном плане отсутствует, добавлен в этап 7.
8. **Публичная страница ученика — токен-ссылка.** Доступ по непугадываемому токену (например, 32 байта), read-only; действия «подтвердить/отменить урок» — отдельные подписанные экшены с rate limiting. Регистрация ученику не нужна (это уже было в плане — фиксируем механику).
9. **Заморозка = статус Enrollment** (`active | paused | archived`), а не отдельная фича. Пауза останавливает генерацию уроков и напоминания.
10. **Дашборд не суммирует валюты.** Доход показывается раздельно по валютам (EUR: …, UAH: …, PLN: …) — без курсов конвертации в MVP.
11. **Soft delete + AuditLog** для студентов, уроков и финансовых записей — преподаватель будет ошибаться, «удалить насовсем» недопустимо.
12. **GDPR-минимум в MVP**: privacy policy, экспорт данных workspace (JSON), полное удаление workspace по запросу, поле «контакт родителя» для несовершеннолетних. DPA и журнал доступа — после выхода на платящих клиентов ЕС.
13. **Тестовая стратегия**: юнит-тесты доменной логики (ledger, cancellation policy, recurrence/DST, снапшот цен) — обязательны и пишутся вместе с кодом; e2e — тонкий smoke-слой позже. Именно доменная логика — конкурентное преимущество, её и тестируем.
14. **Упрощения инфраструктуры для соло-разработчика**: вместо Docker-на-VPS — managed-платформа (Railway/Fly.io: API + Postgres + Redis рядом), web — Vercel. BullMQ в MVP не нужен — хватает `@nestjs/schedule` cron; Redis добавим вместе с онлайн-оплатами. Обязательно: автоматические бэкапы Postgres с первого дня, Sentry.
15. **Монетизация**: в MVP не строим биллинг, но в `Workspace` сразу есть поле `plan` (`free | pro`) и лимиты — чтобы потом не мигрировать.

### Что убрать из исходного списка модулей MVP
`payments-webhooks`, `integrations`, `reports` (дашборда достаточно), `BullMQ/Redis`, push-уведомления (web-MVP: email + copy-to-Telegram текст), Google Calendar sync.

## Архитектура

### Структура монорепозитория (pnpm workspaces + Turborepo)

```
apps/
  web/            — Next.js (App Router, TS, Tailwind, shadcn/ui, TanStack Query,
                    React Hook Form, next-intl, react-big-calendar)
  api/            — NestJS (REST, Swagger/OpenAPI, Prisma, passport-jwt, @nestjs/schedule)
packages/
  domain/         — чистая бизнес-логика без I/O: правила списаний/возвратов,
                    cancellation policy, recurrence/материализация, money-утилиты.
                    Максимальное покрытие тестами (vitest).
  validation/     — Zod-схемы DTO, общие для web и api
  api-client/     — TS-клиент, генерируемый из OpenAPI (openapi-typescript + fetch-обёртка)
  config/         — eslint-config, tsconfig
```

`ui-tokens` — отложен до мобильного приложения.

### Модули NestJS (MVP-набор)

`auth`, `workspaces`, `students` (вкл. parents), `groups`, `enrollments`, `scheduling` (series + lessons + attendance-статусы), `packages` (пакеты + журнал кредитов), `payments` (ручные, + интерфейс PaymentProvider с единственной реализацией `manual`), `notifications` (email + тексты), `dashboard`, `audit`, `public` (страница ученика по токену), `import` (CSV).

### Схема данных (Prisma, ключевые сущности)

- `Workspace` (plan, defaultCurrency, cancellationDeadlineHours)
- `User`, `WorkspaceMember` (role: owner | teacher)
- `Student` (name, contacts, timezone, parentContact?, notes, publicToken)
- `Group`
- `Enrollment` (studentId, groupId?, teacherId, status: active|paused|archived, priceMinor, currency, cancellationDeadlineHours?, billingType: package|monthly|per_lesson)
- `LessonSeries` (enrollmentId|groupId, weekday, localTime, timezone, durationMin, horizonMaterializedUntil)
- `Lesson` (seriesId?, startsAtUtc, status, isDetached, cancelledBy?, cancelledAt?)
- `LessonPackage` (enrollmentId, lessonsTotal, priceMinorSnapshot, currency, purchasedAt, expiresAt?)
- `LessonCreditEntry` (packageId, enrollmentId, delta: ±N, type: purchase|lesson_completed|late_cancellation|teacher_cancellation_refund|manual_adjustment, lessonId?, idempotencyKey unique, note, createdBy)
- `Payment` (enrollmentId, amountMinor, currency, method: cash|bank_transfer|other, packageId?, paidAt, note)
- `AuditLog` (workspaceId, actorId, entity, entityId, action, diff)

Все бизнес-таблицы — с `workspaceId` + составные индексы; `deletedAt` для soft delete.

## План по этапам (соло, ~14–16 недель)

Каждый этап заканчивается задеплоенным на dev-окружение работающим срезом.

**Этап 0. Фундамент (нед. 1–2)**
Монорепо (pnpm + turbo), скелеты api/web, Prisma + Postgres (Railway), auth-заготовка, CI (GitHub Actions: lint + typecheck + test), деплой api → Railway, web → Vercel, Sentry, генерация api-client из Swagger.

**Этап 1. Auth + Workspace (нед. 2–3)**
Email/пароль + JWT (access/refresh), регистрация → автосоздание workspace, роли owner/teacher, i18n-каркас (next-intl, uk + en), базовый layout на shadcn/ui, адаптив с мобильного брейкпоинта.

**Этап 2. Ученики и группы (нед. 4–5)**
CRUD студентов (карточка по списку из исходного плана), группы, enrollments со статусами и индивидуальными правилами (цена, валюта, дедлайн отмены). Настройки workspace (дефолтная валюта, дефолтный дедлайн). Soft delete + AuditLog с этого этапа.

**Этап 3. Расписание (нед. 6–8)** — самый рискованный этап
`packages/domain`: recurrence + материализация с тестами на DST-переходы. `LessonSeries`, cron-материализация на 12 недель, календарь день/неделя/месяц (react-big-calendar), разовые уроки, перенос («только этот / этот и последующие»), статусы урока и state machine переходов. Пока без финансовых последствий — они в этапе 4.

**Этап 4. Пакеты, ledger, оплаты (нед. 9–11)** — ядро продукта
`packages/domain`: правила «переход статуса → операция журнала» с полным покрытием тестами (все 8 сложных кейсов из исходного плана как тест-сценарии: 8 занятий при 9 уроках в месяце, поздняя отмена, возврат при отмене преподавателем, заморозка, снапшот цены и т.д.). LessonPackage, LessonCreditEntry с идемпотентностью, ручные Payment, частичная оплата и долг, ручная корректировка баланса, экран финансов ученика с историей «почему баланс такой».

**Этап 5. Дашборд и напоминания (нед. 12)**
Главный экран: уроки сегодня, «осталось ≤2 занятия», должники, доход за месяц по валютам, отмены. Cron-напоминания на email (Resend), кнопка «скопировать текст напоминания для Telegram».

**Этап 6. Публичная страница ученика (нед. 13)**
Мобильная страница по токен-ссылке: ближайшие уроки, баланс, история оплат, подтверждение/отмена урока (с учётом дедлайна), rate limiting.

**Этап 7. Импорт и пилот SpeakWise (нед. 14–16)**
CSV-импорт студентов/расписания/балансов, перенос реальных данных SpeakWise, полный платёжный месяц через систему, фиксация каждой ручной правки как баг-репорта, багфиксы. GDPR-минимум: privacy policy, экспорт/удаление workspace.

**Вне MVP (следующие этапы, по готовности):** Stripe/WayForPay payment links + webhooks, Google Calendar sync, Telegram-интеграция, Expo-приложение поверх готового API, биллинг подписки SaaS.

## Верификация

- **Юнит-тесты** `packages/domain` (vitest): recurrence/DST, state machine урока, все ledger-правила, идемпотентность, money-утилиты — прогоняются в CI на каждый PR.
- **Интеграционные тесты API** (supertest + тестовый Postgres) на критичные потоки: покупка пакета → проведение урока → списание; отмена преподавателем → возврат.
- **Ручной сценарный чек-лист** по 8 сложным кейсам из позиционирования — прогоняется на dev-окружении в конце этапов 4 и 7.
- **Seed-скрипт** с реалистичным демо-workspace для разработки и демо.
- **Финальная проверка**: месяц реальной работы SpeakWise без ручных правок данных в БД.
